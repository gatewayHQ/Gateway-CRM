Check that src/lib/schema.sql is in sync with how the codebase uses Supabase, then generate a migration SQL script to bring the live database up to date.

## Step 1 — Parse the schema

Read `src/lib/schema.sql`. Extract for each table:
- Column names and types
- Default values
- Check constraints and their allowed values
- Foreign key references

## Step 2 — Audit codebase usage

For each table, grep the source for all INSERT, UPDATE, and SELECT usages:
```
grep -rn "\.from('contacts')\|\.from('properties')\|\.from('deals')\|\.from('tasks')\|\.from('agents')\|\.from('templates')" src/ --include="*.jsx" --include="*.js" -A 5
```

For each usage, extract the column names and values being sent. Then check:

1. **Missing columns**: code uses a column not in `schema.sql` → needs `ALTER TABLE ... ADD COLUMN`
2. **Wrong type**: code sends a number to a text column, or vice versa
3. **Constraint mismatch**: code sends a value not in the `check` constraint list (e.g. a property type that's not in the allowed set)
4. **Missing table**: code calls `.from('table')` on a table not in `schema.sql`

## Step 3 — Check RLS completeness

For each table, verify there's a policy for each required operation:
- SELECT: can agents read rows?
- INSERT: can agents create rows?
- UPDATE: can agents modify rows?
- DELETE: can agents delete rows?

Note any gaps.

## Step 4 — Generate migration SQL

Produce a complete, runnable SQL script:

```sql
-- Gateway CRM — Schema Migration
-- Generated by /schema-sync
-- Run this in Supabase SQL Editor

-- ── New columns ─────────────────────────────────────────────
-- (ALTER TABLE statements for missing columns)

-- ── Constraint updates ──────────────────────────────────────
-- (DROP CONSTRAINT + ADD CONSTRAINT for mismatched check lists)

-- ── New tables ──────────────────────────────────────────────
-- (CREATE TABLE for any table used in code but not in schema)

-- ── RLS policy fixes ────────────────────────────────────────
-- (DROP POLICY / CREATE POLICY for gaps found above)
```

If schema.sql and the codebase are fully in sync, report "No migration needed — schema and codebase are consistent."
